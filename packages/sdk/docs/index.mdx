# STVOR SDK

**Add Signal Protocol encryption to your app with just a few lines of code.**

STVOR implements X3DH key exchange and Double Ratchet encryption (the same cryptography used by Signal) using **only Node.js built-in `crypto` module** — zero external dependencies. Add world-class end-to-end encryption without being a cryptographer.

```typescript
import { Stvor } from '@stvor/sdk';

// One call: generates keys, registers with relay, starts polling
const alice = await Stvor.connect({
  userId:   'alice@example.com',
  appToken: 'stvor_live_...',
  relayUrl: 'https://relay.stvor.xyz',
});

// Send encrypted messages — E2EE happens automatically
await alice.send('bob@example.com', 'Hello, Bob!');

// Receive messages
alice.onMessage((msg) => {
  console.log(`${msg.from}: ${msg.data}`);
});
```

<Tip>
**Zero cryptography knowledge required.** The SDK handles X3DH handshakes, key rotation, and Double Ratchet encryption automatically.
</Tip>

## Quickstart

Get up and running in under 30 seconds:

### 1. Install

```bash
npm install @stvor/sdk
```

### 2. Start the mock relay (for local development)

```bash
npx @stvor/sdk mock-relay
```

The mock relay lets you test E2EE without production credentials.

### 3. Send your first encrypted message

```typescript
import { Stvor } from '@stvor/sdk';

const alice = await Stvor.connect({
  userId:   'alice@example.com',
  appToken: 'stvor_dev_test123',      // any stvor_* token works locally
  relayUrl: 'http://localhost:4444',
});

const bob = await Stvor.connect({
  userId:   'bob@example.com',
  appToken: 'stvor_dev_test123',
  relayUrl: 'http://localhost:4444',
});

// Receive messages
bob.onMessage((msg) => {
  console.log(`${msg.from}: ${msg.data}`);
  // "alice@example.com: Hello from Alice!"
});

// Send an encrypted message
await alice.send('bob@example.com', 'Hello from Alice!');

// Disconnect when done
await alice.disconnect();
await bob.disconnect();
```

<Warning>
**Never commit tokens to version control.** Use environment variables:
```typescript
const alice = await Stvor.connect({
  userId:   'alice@example.com',
  appToken: process.env.STVOR_APP_TOKEN,
  relayUrl: process.env.STVOR_RELAY_URL,
});
```
</Warning>

# SDK Basics

## Connecting a User

```typescript
const client = await Stvor.connect(config);
```

Connects a user, generates their identity keys, and registers them with the relay — all in one call.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `config.userId` | `string` | Yes | User identifier (email, username, UUID) |
| `config.appToken` | `string` | Yes | Token from developer dashboard |
| `config.relayUrl` | `string` | Yes | Relay server URL |
| `config.timeout` | `number` | No | Request timeout ms (default: 10000) |
| `config.pollIntervalMs` | `number` | No | Message polling interval ms (default: 1000) |

| Returns | Description |
|---------|-------------|
| `Promise<StvorClient>` | Client instance for this user |

```typescript
const alice = await Stvor.connect({
  userId:   'alice@example.com',
  appToken: process.env.STVOR_APP_TOKEN,
  relayUrl: 'https://relay.stvor.xyz',
});

// Get user ID
alice.getUserId(); // 'alice@example.com'
```

## Sending Messages

```typescript
await client.send(recipientId, data, options?);
```

Sends an encrypted message. **Automatically waits** for the recipient to register (configurable).

| Parameter | Type | Description |
|-----------|------|-------------|
| `recipientId` | `string` | Recipient's user ID |
| `data` | any | Message data — any JavaScript type (see Data Types table) |
| `options.waitForRecipient` | `boolean` | Auto-wait for recipient (default: true) |
| `options.timeout` | `number` | Max wait for recipient ms (default: 10000) |

```typescript
// Send text
await alice.send('bob@example.com', 'Hello!');

// Send with custom timeout
await alice.send('bob@example.com', 'Hello!', { timeout: 30000 });

// Skip waiting — throws immediately if recipient not found
await alice.send('bob@example.com', 'Hello!', { waitForRecipient: false });
```

## Supported Data Types

The SDK preserves the exact JavaScript type — no data is lost through serialization.

| Type | Example |
|------|---------|
| `string` | `'Hello, world!'` |
| `number` | `42`, `3.14`, `-7` |
| `boolean` | `true`, `false` |
| `null` | `null` |
| `Uint8Array` / `Buffer` | Binary files, images, audio |
| `object` / `array` | `{ key: 'value' }`, `[1, 2, 3]` |
| `Date` | `new Date()` |
| `Set` | `new Set([1, 2, 3])` |
| `Map` | `new Map([['a', 1]])` |

```typescript
// All of these work out of the box:
await alice.send('bob', 'text message');
await alice.send('bob', 42);
await alice.send('bob', true);
await alice.send('bob', null);
await alice.send('bob', new Uint8Array([1, 2, 3]));
await alice.send('bob', { user: 'alice', score: 100 });
await alice.send('bob', new Date());
await alice.send('bob', new Set(['a', 'b', 'c']));
await alice.send('bob', new Map([['key', 'value']]));

// Recipient gets the original type back:
client.onMessage((msg) => {
  // msg.data is the exact type that was sent
  if (msg.data instanceof Date) {
    console.log('Received date:', msg.data.toISOString());
  }
});
```

## Receiving Messages

```typescript
const unsubscribe = client.onMessage(handler);
```

Subscribes to incoming messages. Messages arrive via background polling.

**Node.js SDK:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `handler` | `(msg: StvorMessage) => void` | Called for each message |

```typescript
interface StvorMessage {
  from:      string;   // Sender's user ID
  data:      unknown;  // Original type preserved
  timestamp: Date;
  id:        string;   // Message identifier
}

const unsubscribe = bob.onMessage((msg) => {
  console.log(`${msg.from}: ${msg.data}`);
});

unsubscribe(); // Stop receiving
```

**Browser SDK:**

```typescript
// Browser handler receives (from, data) separately
alice.onMessage((from, data) => {
  console.log(from, data);
});
```

## Disconnecting

```typescript
await alice.disconnect();
```

Stops message polling and closes the relay connection. Call when the user logs out or the app closes.

# API Reference

## `Stvor.connect()` — Node.js

```typescript
function connect(config: StvorConfig): Promise<StvorClient>
```

Single-call API: creates identity keys, registers with relay, starts polling.

### Parameters

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `config.userId` | `string` | Yes | User identifier |
| `config.appToken` | `string` | Yes | Token from developer dashboard (must start with `stvor_`) |
| `config.relayUrl` | `string` | Yes | Relay URL |
| `config.timeout` | `number` | No | Timeout in ms (default: 10000) |
| `config.pollIntervalMs` | `number` | No | Polling interval in ms (default: 1000) |

### Returns

`Promise<StvorClient>`

### Example

```typescript
import { Stvor } from '@stvor/sdk';

const alice = await Stvor.connect({
  userId:   'alice@example.com',
  appToken: process.env.STVOR_APP_TOKEN,

});
```

### Errors

- `INVALID_APP_TOKEN` — Token doesn't start with `stvor_`
- `RELAY_UNAVAILABLE` — Cannot connect to relay

# Cryptography

STVOR v3.3 implements the **Signal Protocol** — zero external crypto libraries.

**Node.js SDK** uses the built-in `node:crypto` module:
- **X3DH** (Extended Triple Diffie-Hellman) for initial key agreement
- **Double Ratchet** for forward secrecy and post-compromise security
- **ECDH P-256** for Diffie-Hellman key exchange
- **ECDSA P-256** for digital signatures (SPK verification)
- **AES-256-GCM** for authenticated encryption (header as AAD)
- **HKDF-SHA256** for key derivation
- **HMAC-SHA256** for chain key ratcheting

**Browser SDK** uses the Web Crypto API — identical security, no Node.js, no WASM.

## Security Guarantees

✅ **End-to-end encryption** — Plaintext never leaves the device  
✅ **Forward Secrecy** — DH key rotation every message  
✅ **Post-Compromise Security** — Automatic DH ratchet advancement  
✅ **Authenticated encryption** — AES-256-GCM with AEAD  
✅ **Replay protection** — Nonce + timestamp validation  
✅ **TOFU fingerprint verification** — Identity verification on first contact  
✅ **Simultaneous-send support** — Both parties can send before receiving  

## What STVOR Does NOT Guarantee

❌ **Metadata privacy** — Relay sees sender, recipient, message timing  
❌ **Strong MITM after first contact** — TOFU model (first-contact trust is user responsibility)  
❌ **Multi-device sync** — Each device = separate identity  
❌ **Post-quantum security** — Uses classical P-256 (planned: ML-KEM hybrid)  
❌ **Persistent storage** — Keys/sessions stored in-memory (restart clears state)  

## Threat Model

**Attacker capabilities:**
- Network eavesdropper (passive)
- Relay operator (honest-but-curious)
- Active MITM on first contact (TOFU vulnerability)

**What attacker CANNOT do:**
- Decrypt message content (without private keys)
- Forge authenticated messages
- Replay previously delivered messages

**Out of scope:**
- Device compromise (malware, root access)
- Social engineering
- Screen capture / screenshots

<Warning>
**First-contact trust**: TOFU model means first contact requires user verification. After first contact, strong MITM protection is active.
</Warning>

# FAQ

**Q: Do I need to understand cryptography?**

No. X3DH handshake and Double Ratchet are fully automatic. Keys are generated on `connect()` and rotated transparently.

**Q: Is this really end-to-end encrypted?**

Yes. Plaintext never leaves the device. The relay only forwards ciphertext and cannot decrypt messages.

**Q: What data types can I send?**

Any JavaScript value: strings, numbers, booleans, null, binary (Uint8Array/Buffer), objects, arrays, Date, Set, and Map. The original type is preserved on the receiving end.

**Q: Can users have multiple devices?**

Not yet. Each device is a separate identity. Multi-device sync is planned for a future version.

**Q: What happens if the server restarts?**

Keys, sessions, TOFU fingerprints, and the replay cache are cleared (in-memory storage). Clients need to reconnect.

**Q: Can users recover messages on a new device?**

No. Storing keys server-side would break E2EE. Forward secrecy means past messages are unrecoverable by design.

**Q: Is STVOR post-quantum secure?**

No. v3.3 uses classical P-256 (ECDH/ECDSA). Hybrid ML-KEM-768 is planned but not yet implemented.

**Q: Can I use this in the browser?**

Yes. Import from `@stvor/sdk/web` and use `StvorWebSDK.create()`. It uses the browser's native Web Crypto API with zero external dependencies.

## Troubleshooting

**`INVALID_APP_TOKEN`** — Your token must start with `stvor_`. Use `stvor_dev_test123` for local development.

**`RELAY_UNAVAILABLE`** — Make sure the mock relay is running (`npx @stvor/sdk mock-relay`) or the relay URL is correct.

**`RECIPIENT_NOT_FOUND`** — The recipient hasn't connected yet. Either wait longer, or use the default `waitForRecipient: true`.

**`DELIVERY_FAILED`** — Decryption failed on receive. Usually means a session state mismatch — disconnect and reconnect both clients.

**`ERR_PACKAGE_PATH_NOT_EXPORTED`** — Make sure you're importing from the correct path: `@stvor/sdk` for Node.js, `@stvor/sdk/web` for browser.
